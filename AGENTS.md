# AGENTS.md
Guidance for AI coding agents (Claude Code, Codex) working in this repository.

## What this is

An unofficial **Travian T4** private-server clone (browser strategy game), forked from TravianX/ZravianX, with custom UI/gameplay changes and a Spanish translation. Legacy PHP `7.4` + Apache + MariaDB `10.6`, all provided via Docker. This is old procedural PHP with a thin class layer — expect string-concatenated SQL, `global` singletons, and no autoloader/framework.

## Running the project

Everything runs through Docker Compose (stack name `traviant4-php7`). The repo is bind-mounted into the web container, so **PHP edits are live with no rebuild** — only rebuild after changing `docker/Dockerfile` or `docker/php.ini`.

```bash
docker compose up --build --detach   # start / rebuild
docker compose ps                     # status (db must be "healthy")
docker compose logs --follow web db   # logs
docker compose down                   # stop, keep dbdata volume
./deploy.sh                           # git pull + rebuild (used on the prod box)
```

- Game: <http://localhost:8080/> · Installer: <http://localhost:8080/install/> · Admin: <http://localhost:8080/Admin/>
- MariaDB from host: `127.0.0.1:3308` (db `travian_t4`, user `travian`). Inside Docker the app connects to host `db` port `3306`.
- `reset_server.sh` wipes and reinstalls the game world from scratch (destructive).
- There is **no test framework and no linter**. `test.php`/`test22.php` are ad-hoc scratch scripts; `tools/check_quest_rewards.php` is a one-off checker. Verify changes by loading pages in the running container.

## First-time setup / DB schema

The DB schema and config are **generated by the web installer** at `/install/`, not committed. It writes `config/connection.php` (DB creds, gitignored), `config/config.php`, and `config/installed`. SQL schema changes live in `tools/migrations.sql`. MariaDB runs with `--sql-mode=` (non-strict) on purpose — the legacy schema relies on zero dates (`'0000-00-00'`) and loose inserts, so don't assume strict-mode behavior.

## Architecture

### Request flow
Each top-level `*.php` in the repo root is a full page (`dorf1.php` = village resource view, `dorf2.php` = village center/buildings, `dorf3.php`, `build.php` = single building, `hero*.php`, `karte.php` = map, `berichte.php` = reports, `nachrichten.php` = messages, `allianz.php`, `spieler.php` = profile, etc.). German-named files are a leftover of the Travian origin. `ajax.php` handles AJAX (`?f=` / `?cmd=` switch), `do.php` redirects, `login.php`/`anmelden.php` handle auth.

A game page typically: `include("GameEngine/Village.php")` → run game logic → then include a chain of `Templates/*.tpl` partials (`html.tpl`, `field.tpl`, `movement.tpl`, `production.tpl`, `troops.tpl`, `sideinfo.tpl`, `footer.tpl`, `header.tpl`, `res.tpl`, `vname.tpl`, `quest.tpl`).

### Bootstrap & global singletons
There is no DI. Including `GameEngine/Village.php` pulls in `Session.php`, which chains includes of `config/connection.php`, `config/config.php`, `Database.php`, the `Data/*` game tables, `Lang/<LANG>.php`, and the other engine classes. Several **global objects are instantiated at include time** and used everywhere via `global $...`:

- `$database` — `GameEngine/Database/db_MYSQLi.php` (`class mysqli_DB`, hundreds of query methods)
- `$session` — `GameEngine/Session.php` (login state, current village `$_SESSION['wid']`)
- `$generator` — `GameEngine/GeneratorX.php`
- `$logging` — `GameEngine/Logging.php`

When adding logic, follow this pattern (`global $database, $session, ...`) rather than introducing new instantiation.

### Database layer
`GameEngine/Database.php` switches on `DB_TYPE` (set in `connection.php`, `1` = MySQLi) and includes `Database/db_MYSQLi.php`. That file defines the `mysqli_DB` class **and**, at the bottom, legacy `mysql_query()` / `mysql_fetch_assoc()` / `mysql_num_rows()` / `mysql_real_escape_string()` / `mysql_error()` shim functions mapping the old `mysql_*` API onto the shared `$database->connection`. Legacy code across the repo calls these bare `mysql_*` functions directly — that's expected, not a bug.

### Config constants
`config/config.php` reads one row from the `config` DB table into `$result` and `define()`s the server constants used throughout the code: `SERVER_NAME`, `SPEED`, `LANG`, `WORLD_MAX`, `HOMEPAGE`/`SERVER` (public URL), `TB_PREFIX` (table prefix, e.g. `s1_`), protection timers, etc. Table names are always built as `TB_PREFIX . "tablename"`.

### Game data & logic
`GameEngine/Data/*.php` are static lookup tables (building costs `buidata`, units `unitdata`, resource fields `resdata`, hero `hero_full`, culture points `cp`, celebrations `cel`). Core mechanics: `Village.php`, `Building.php`, `Market.php`, `Technology.php`, `Battle.php`, `Account.php`, `Alliance.php`, `Ranking.php`, `Message.php`, `Inventory.php`, `Units.php`, `Profile.php`. `Automation.php` / `AutomationSinglePlayer.php` drive automated/bot behavior.

### Templates
`.tpl` files are **plain PHP includes**, not a template engine — they mix HTML and `<?php ?>` and read the same globals. Edit them like PHP.

### Localization
`GameEngine/Lang/{es,en,de,ru,fa}.php` each define UI text as `define()` constants plus a `$lang` array. `es.php` (Spanish) is the active target locale; translation from the English original is an **ongoing effort** — player-facing pages are done, the Admin panel is the main remaining English area. Use neutral/LatAm Spanish, not peninsular terms.

### Admin panel
The **live** admin panel is `Admin/admin.php` → `Admin/Templates/`. `Admin/tpl/` and `Templates/Admin/` are dead copies — don't edit those expecting changes to show.

## Conventions & gotchas

- **Timezone is hard-coded** `America/Argentina/Buenos_Aires` (in `Village.php` and docker-compose), matching the Docker `TZ`.
- The web container runs Apache as **UID 1000** so it can write the bind-mounted code and `config/` / `install/installconfig/`. Permission errors on those paths usually trace back to a UID mismatch.
- `config/connection.php`, `config/installed`, `.env`, and `backup-*.sql` are gitignored; never commit generated config or secrets.
- Do **not** touch any sibling `travian-*` Docker containers on this host — only the `traviant4-php7` stack belongs to this project.
- Sidebar layout is fragile: never widen `#side_info` / `#villageList` in `compact1.css` (breaks the float layout); grow vertically instead.
- **Commit style:** the commit must be **only a title** — short, in English, no body/description. The author is always the repo user (Matias); **never add a `Co-Authored-By` trailer or any other co-author**.

## Deployment note

Production is **travian.chewer.net**, served from a **Raspberry Pi 4B (8GB RAM)**, separate from local Docker. **Every push to git triggers `deploy.sh` on the Pi** (`git pull` + `docker compose up --build --detach`), so pushed changes go live automatically — treat `origin/master` as production. The redeploy is **near-instant**: the compose stack is just MariaDB + PHP and the game source is bind-mounted as a volume, so PHP/template edits apply immediately and the image only actually rebuilds when `docker/Dockerfile` or `docker/php.ini` change (rare, and the only case where the ARM/8GB slowness matters).

Schema changes that the installer would normally create must be applied to the prod DB manually (see `tools/migrations.sql`).
